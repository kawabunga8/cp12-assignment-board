GitHub DevOps Workflow
Prototype & TestMaking a change on a branch, testing in Unity, and iterating before submitting the PR is a direct prototype-and-test loop. MakeIndependently editing a script, committing, downloading, and verifying in Unity demonstrates technical execution with real tools. ShareThe Pull Request is the share artifact: a professional title, description of what was changed, and how it was verified.
Practice a professional DevOps workflow using GitHub Online and Unity. This process emphasizes collaboration, relationship, and technical fluency. You will make one improvement to a shared codebase and submit it for review.
Mission Steps
1
Fork the Repo — Your Cloud Copy
Open github.com/RCSCP12/MusicNoteGame and click Fork (top-right).
  • Check the URL: it must say github.com/YOUR-USERNAME/MusicNoteGame
  • If it still says RCSCP12, you are on the teacher's copy and cannot save changes!
2
Create a Feature Branch — Your Sandbox
Always work on a branch so the "Main" version stays safe if something breaks.
  • Click the button that says main (top-left of the file list)
  • Type a descriptive name, e.g. update-menu-colors or fix-scoring-logic
  • Click Create branch: [your-branch-name]
3
Edit & Commit — Saving Online
Navigate to the script you want to change, click the pencil icon, make your edit, then commit.
  • Navigate to e.g. Assets/Scripts/MenuManager.cs
  • Click the Pencil Icon (top-right) to open the web editor
  • Make your ONE change
  • Click the green Commit changes button
  • Type a short note about what you did, then click Commit again
  • If you don't commit, your work isn't saved!
4
Test in Unity — The Reality Check
Download a fresh ZIP from your branch and open it in Unity 6 (6000.3.3f1).
  • Delete any old MusicNoteGame folders from your computer first
  • Click the green <> Code button → Download ZIP
  • Unzip the folder to your desktop
  • Open Unity Hub → Add > Add project from disk → select the unzipped folder
  • Press Play and verify your change works
  • If it doesn't work, return to Step 3, fix it, and download a new ZIP
5
Submit a Pull Request
On your GitHub page, click Compare & pull request and fill in the details.
  • Title: Use a professional name, e.g. Feature: Added High Score Tracker
  • Description: Explain what you changed and how you verified it in Unity
  • Click Create pull request
🛠 Troubleshooting
"I don't see my changes in Unity."You likely opened an old version. Delete all local copies, download a fresh ZIP from your branch, and re-open in Unity Hub.
"Everything is pink or missing."Wrong Unity version. Make sure you are using Unity 6 (6000.3.3f1).
"I can't find the Commit button."You need to type an actual change in the editor first — the button stays hidden until a modification is made.
"My PR was rejected."Check the teacher's comments! Return to Step 3, fix the issue, and update your branch.

Lab 2 – Project Primitive
Due 30 January 2026 — 15:00 · Setting the Scene
Define & IdeateTranslating your Lab 1 MVP sketch into 3D means making active design decisions about player, enemies, environment, and camera — design thinking in action. Prototype & TestBuilding a greybox specifically to test layout and scale before committing to final visuals is the core purpose of prototyping. The video submission is evidence of testing. MakeBuilding with primitives, applying materials, setting the camera, and saving a Unity package all demonstrate technical execution and independent production skills.
Turn your Lab 1 sketch into a 3D greybox prototype in Unity using simple shapes. Test layout and scale before adding graphics. This lab is about working smart, not looking polished.
Steps
1
Review Your Plan
Look at the MVP Sketch from Lab 1 before you start building.
2
Build with Primitives
Use Unity primitive shapes to represent player, enemies, obstacles, and environment. No detailed models.
  • Available shapes: Cube, Sphere, Capsule, Plane
  • Represent: player, enemies, obstacles, environment
  • Do not import detailed models at this stage
3
Apply Colors (Materials)
Give objects distinct colors so they are easy to identify — e.g., player = blue, enemies = red, ground = green.
4
Set the Camera
Position the Main Camera to match your game type: top-down, side-view, or third-person.
5
Check Layout
Confirm all key elements from your Lab 1 sketch are present and readable in the scene.
6
Save Your Scene
Export a Unity package and save it with your name and version number.
  • In Project window, right-click Assets → Export Package
  • Click Export
  • Create a Backups folder in your Personal Project folder
  • Save with your name and version, e.g. Carl_V0.1.unitypackage
Deliverable
  • Short video showing the scene and camera view
  • Must clearly show: primitive shapes, coloured materials, correct camera view
Grading Criteria
CategoryProficient (Target)
Prototyping StrategyUses simple primitives to test layout without focusing on final visuals.
Technical SkillsObjects are clearly coloured and the camera is correctly positioned.
Design ImplementationThe 3D scene accurately reflects the Lab 1 sketch and design intent.
Extending: Design is refined after viewing it in 3D, with clear justification.  ·  Developing/Emerging: Scene is unclear, incomplete, or overly focused on visuals.
